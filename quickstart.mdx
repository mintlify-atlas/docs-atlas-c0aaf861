---
title: "Get started with Lucy"
description: "Install Lucy and manage your first Minecraft server in under 5 minutes. Covers installation, init, adding packages, and checking server status."
---

Lucy is a CLI tool written in Go. This guide walks you through installing Lucy, initializing it in a server directory, adding packages, and applying your first sync — whether you are starting fresh or adopting an existing server.

<Steps>
  <Step title="Install Lucy">
    Lucy is distributed as a Go binary. Install it with:

    ```bash
    go install github.com/mclucy/lucy@latest
    ```

    This requires **Go 1.21 or later**. After installation, verify it is on your path:

    ```bash
    lucy --help
    ```
  </Step>

  <Step title="Create or navigate to your server directory">
    Lucy works on both fresh directories and existing servers.

    **Fresh server:**

    ```bash
    mkdir my-server
    cd my-server
    ```

    **Existing server:**

    ```bash
    cd /path/to/your/server
    ```

    <Tip>
      Lucy is designed for existing servers. If you already have mods, plugins, or a server JAR in place, `lucy init` will discover them rather than replace them.
    </Tip>
  </Step>

  <Step title="Initialize Lucy">
    Run `lucy init` to take over the current directory:

    ```bash
    lucy init
    ```

    Lucy inspects the directory, detects the game version, server core, and any installed mods or plugins, and then walks you through an interactive flow to define what it should manage. Nothing is written until you confirm.

    After confirming, Lucy creates three files:

    ```text
    .lucy/
      config.toml     # Scope, source preferences, and optional settings
      manifest.json   # Your package intent (what you want installed)
      lock.json       # Exact resolved facts (versions, hashes, paths)
    ```

    For a non-interactive init that accepts all defaults, use:

    ```bash
    lucy init --yes --game-version 1.21.4
    ```

    If a `.lucy/` directory already exists, Lucy preserves it by default. Use `--conflict overwrite` to replace it or `--conflict abort` to stop on conflict.
  </Step>

  <Step title="Add packages">
    Add mods, plugins, or server cores to your manifest with `lucy add`. Packages are identified using the format `platform/name@version`:

    ```bash
    lucy add fabric/fabric-api
    ```

    ```bash
    lucy add neoforge/create-aeronautics@latest
    ```

    ```bash
    lucy add mcdr/example-plugin@compatible
    ```

    - **`platform`** — the modding platform or runtime the package targets (e.g., `fabric`, `neoforge`, `mcdr`). Omit it and Lucy infers from the detected environment.
    - **`name`** — the mod, plugin, or project name or ID.
    - **`@version`** — a specific version, `@latest` for the newest available, or `@compatible` (the default) for the newest version that fits the current environment.

    Lucy resolves transitive dependencies automatically and updates both `manifest.json` and `lock.json`.
  </Step>

  <Step title="Check server status">
    Run `lucy status` at any time to get a `neofetch`-style overview of the current server environment:

    ```bash
    lucy status
    ```

    The output includes game version, server core, modding platform, detected environment topology, runtime activity, risk signals, and a list of installed mods or plugins.

    Use `--long` to see full package identifiers and install paths, or `--json` for machine-readable output:

    ```bash
    lucy status --long
    lucy status --json
    ```
  </Step>

  <Step title="Sync the server">
    Apply the lockfile to the managed runtime scope:

    ```bash
    lucy install
    ```

    When the lockfile is current and matches the manifest, `lucy install` uses exact locked versions and hashes. If the lock is stale or missing, it falls back to resolving from manifest intent. All operations run independently of the server runtime — you can sync while the server is stopped.
  </Step>
</Steps>
